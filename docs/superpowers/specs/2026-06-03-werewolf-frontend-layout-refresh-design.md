## 目标

为当前前端仓库的现有单页狼人杀界面做一次布局与视觉重构。

这次变更不追求完整复刻参考项目，也不扩展后端能力。目标是：

1. 参考 `E:\VSCode\golang\Werewolf` 的视觉布局与信息层级。
2. 保持当前仓库 `src/` 下 React 单页结构与现有后端联调能力。
3. 第一版同时覆盖两种核心页面状态：`未开局页` 与 `游戏主界面`。
4. 默认优先旁观视角，但版式要为未来“玩家 / 旁观”切换预留空间。

## 范围

本次只处理以下范围：

- `未开局页`
- `游戏主界面`
- 现有游戏主界面的骨架布局、信息层级、视觉主题
- 与现有后端接口一致的交互状态展示

明确不在本次范围内：

- 新增后端接口
- 新增路由或多页导航
- 房间系统、账户系统、房主逻辑
- 真实的玩家视角隔离
- 本地前端状态机替代服务端状态

## 采用方案

采用 `B. 轻量结构重组` 方案。

即：保留当前单页状态切换与现有 API 流程，不推翻 `useGameConsole` 主数据流；但重排主界面骨架、收紧组件职责，并将视觉方向调整为“现代可读性 + 狼人杀主题识别”的折中路线。

这个方案优于纯样式换皮，因为当前组件的信息顺序并不完全匹配目标布局；也优于提前产品化扩展，因为现有后端尚不足以支撑更完整的视角模式和玩法差异。

## 页面架构

应用继续保持单入口、双状态页面：

```text
App
├─ PreGameScreen
└─ GameExperience
   ├─ TopBanner
   ├─ MainStage
   │  ├─ NarrativeLog
   │  └─ SideRail
   │     ├─ ControlPanel
   │     ├─ GameSummary
   │     └─ StatusStrip
   └─ SituationZone
      ├─ PlayerGrid
      └─ RoleSpotlight
```

状态切换规则保持简单：

- 游戏未初始化时进入 `PreGameScreen`
- 游戏已初始化时进入 `GameExperience`

不引入第三种本地“过渡路由态”。视觉切换由服务端真实状态驱动。

## 页面信息分区

### 1. 未开局页

未开局页承担三件事：

1. 提供进入牌局的舞台感。
2. 清楚展示服务是否可用。
3. 给出单一且明确的开始动作。

页面结构：

- 左侧主视觉区：标题、简短氛围文案、版本定位说明
- 右侧服务面板：健康状态、启动说明、错误提示
- 底部或附属说明卡：告诉用户开局后会进入实时观战控制台

未开局页不展示后端当前不支持的角色预选、房间选择或本地规则配置。

### 2. 游戏主界面

游戏主界面采用“模块折中”骨架：

1. 顶部横幅区
2. 中段主内容区
3. 底部态势区

其中：

- 顶部横幅区负责阶段、回合、胜负和氛围定位，是整页视觉锚点。
- 中段左侧为主叙事日志，右侧为操作与摘要信息。
- 底部区域为玩家态势与角色聚焦。

该骨架默认优先旁观视角，但不封死未来的玩家视角切换。

## 组件职责

### App

`App.tsx` 继续只负责页面状态分流：

- 根据 `useGameConsole` 的 `viewModel.screenMode` 决定当前页面
- 不回收布局细节，不重新膨胀成大而杂的入口文件

### PreGameScreen

`PreGameScreen` 继续独立存在，但升级成完整开场舞台：

- 氛围引导
- 服务状态检查
- 启动动作入口
- 错误与说明信息

### GameShell

`GameShell` 从“通用四槽容器”调整成更明确的布局骨架容器，负责：

- 顶部横幅区
- 中段双栏区
- 底部态势区

它承接页面结构，不承接具体业务判断。

### PhaseHero

`PhaseHero` 升级为顶部横幅核心组件，负责展示：

- 当前回合
- 当前昼夜阶段
- 胜负/结束状态
- 简短氛围文案或阶段提示

它应从局部标题块升级为全局信息锚点。

### NarrativeLog

`NarrativeLog` 升级为主舞台组件。

它在视觉上成为主屏，承担“观战时最先阅读的区域”，而不只是右侧附属信息。

### ControlPanel

`ControlPanel` 继续负责：

- 刷新状态
- 开始游戏
- 推进阶段

但视觉表达要更接近“导演台 / 控制台”，而不是普通按钮集合。

### GameSummary 与 StatusStrip

右栏信息需要收紧：

- `GameSummary` 负责局势摘要
- `StatusStrip` 收缩成异常提示与请求错误提示

避免右侧每块都同权，造成碎片化阅读。

### PlayerGrid

`PlayerGrid` 继续作为底部玩家态势区，但要强化以下视觉区分：

- 存活 / 死亡
- 当前焦点
- 关键角色或关键状态摘要

目标是让用户能快速扫读牌桌态势，而不是只看到一组均质卡片。

### RoleSpotlight

新增轻量的“角色聚焦”组件。

第一版不引入真实视角逻辑，只根据现有 `state + messages` 推导当前重点，例如：

- 上一夜结果
- 当前阶段关键角色
- 当前回合焦点玩家

它的主要作用是为未来模式切换和个体视角留出自然位置。

## 数据与状态流

继续沿用现有三层结构：

1. `api.ts`：封装 `/api/game/*` 请求
2. `useGameConsole` / `view-model`：协调页面状态、请求状态、推导展示字段
3. 展示组件：只消费视图模型结果

可用接口范围固定为：

- `POST /api/game/start`
- `POST /api/game/next`
- `GET /api/game/state`
- `GET /api/game/messages`
- `GET /api/game/health`

设计要求：

- 不新增前端伪状态机
- 不为了布局新增后端字段依赖
- 新的摘要区与角色聚焦区优先由现有状态推导

## 异常与状态处理

### 未初始化

后端返回未初始化状态时，明确进入 `未开局页`，不视为错误。

### 服务异常

服务异常应在两种页面都可见：

- 未开局页：可以更完整提示
- 游戏主界面：只做轻提示，不抢走主叙事区域

### 请求中

以下动作都必须有明确加载反馈：

- `开始游戏`
- `推进阶段`
- `刷新状态`

避免页面“没有动静但用户不知道是否正在请求”的情况。

### 游戏结束

结束态需要统一收束：

- 顶部横幅进入结算态
- 摘要区展示胜负结果
- 玩家区体现终局状态

## 视觉语言

采用“现代与主题折中”的视觉方向。

### 字体

- 标题使用更有戏剧感的展示字体或更强字重
- 正文保持高可读性
- 不走纯系统工具风，也不走夸张恐怖海报风

### 配色

- 主背景以深夜蓝黑系为主
- 酒红、琥珀色只作为重点高亮
- 不使用高饱和整页紫红噪音

### 卡片语言

- 统一大圆角卡片
- 半透明分层
- 柔和描边与轻微发光
- 通过信息密度区分层级，而不是让每个模块都争抢焦点

### 氛围背景

- 使用渐变、局部光晕、轻纹理
- 未开局页氛围更强
- 游戏主界面更克制、更偏实时控制室

## 扩展边界

这次实现必须为后续扩展留口，但不预做过量能力。

允许未来扩展：

- 玩家 / 旁观视角切换
- 房间态信息
- 更多操作面板

本次不实现：

- 真实视角权限差异
- 多页面产品化信息架构
- 账户与房间流转

## 验证策略

至少覆盖以下验证：

- `npm run typecheck`
- `npm run test`
- `npm run build`

补充的关键测试优先级：

1. 未开局页状态展示
2. 主界面骨架渲染
3. 错误态与结束态展示
4. 若调整 `view-model` 推导逻辑，则补对应行为测试

## 风险与控制

本次最大风险不是样式细节，而是组件职责再次发散，导致入口文件回到臃肿状态。

控制方式：

- 入口只分流
- 骨架容器负责布局
- 展示组件各守边界
- 数据推导尽量留在现有 `view-model` 体系内

## 实施结论

本次前端应被实现为：

- 一个继续由后端真实状态驱动的单页应用
- 两种核心页面状态：未开局页、游戏主界面
- 一套更接近参考项目气质、但不照抄旧交互的视觉系统
- 一个默认偏旁观视角、同时为未来玩家视角预留空间的模块化骨架
