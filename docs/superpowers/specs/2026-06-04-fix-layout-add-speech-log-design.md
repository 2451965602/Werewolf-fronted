## Summary

本次变更把观战控制台从“顶部长 banner + 中段双列 + 下段局势区”的松散组合，收敛成桌面端优先的中心舞台布局。中央继续保留叙事实录时间线；顶部大 banner 压缩为紧凑状态条；席位信息改为围绕时间线的环形席位略缩图；当前回合的玩家发言记录独立放在远离时间线的一侧，作为同步视图快速回看本轮发言。

这次变更只主动优化桌面端布局。移动端不引入新的圆桌排布，但实现时必须保持现有纵向退化布局可用，避免回归。

## Problem

当前桌面端控制台在 `GameShell` 中把 `banner`、双列内容区和 `situation` 作为三个纵向兄弟节点塞进固定屏高 `main`。由于中段双列区被设置为 `flex-1`，而下方 `situation` 仍按内容自然撑开，最终会出现中段区块在桌面端被压缩为零高度的情况，内部内容只能依赖 `overflow` 被“硬挤”出来，导致滚动和交互都不稳定。

信息结构也不够聚焦：

- 顶部 hero 占据了大量垂直空间，却只承载阶段性状态。
- 详细席位大卡片与主时间线争夺视觉注意力。
- 发言虽然已经存在于 `messages` 和 `timeline` 中，但用户无法快速回看“当前回合的玩家发言集合”。

## Goals

- 修复桌面端控制台布局约束，让主要区域具备稳定高度和可预测滚动。
- 保留叙事实录时间线作为唯一主舞台。
- 新增与时间线同步的“当前回合发言记录”独立视图。
- 用环形席位略缩图承载“几号、什么身份、是否死亡”的快速态势感知。

## Non-Goals

- 不改动后端消息生成逻辑。
- 不新增任何消息写入接口或手动录入能力。
- 不为移动端设计新的圆桌布局。
- 不保留“现有完整玩家卡片围成一圈”的实现目标；该形式在信息密度和响应式上都不可控。

## Proposed UX

### 1. 顶部状态条取代大 banner

原有 `PhaseHero` 从大体积主视觉收缩为紧凑状态区，仅保留：

- 当前轮次
- 当前阶段
- 牌局状态 / 胜负状态
- 最近系统播报摘要（可选）

理由：阶段状态是持续上下文，不适合做成弹窗；弹窗会引入新的显示状态机、遮挡主舞台，也不利于浏览历史。系统播报本身已经有时间线承载，更合理的做法是把状态区用于“现在是什么局面”，而不是重复讲完整故事。

### 2. 中央叙事舞台保持时间线

中央区域保留 `NarrativeLog`，继续展示系统、投票、玩家发言的混合叙事流。该区仍负责：

- 展示完整消息顺序
- 高亮最新相关消息
- 支持历史记录滚动浏览

它仍是页面视觉中心，不与发言记录角色重叠。

### 3. 环形席位略缩图围绕主舞台

“圆桌会议”概念的可行实现，不是把现有大卡片围成一圈，而是新增一个紧凑的席位可视化层：

- 每个席位节点只显示席位号
- 节点内部或下沿显示身份标记
- 死亡玩家使用置灰、透明度降低、边框弱化等方式处理
- 活着的玩家保持高亮状态

这块更像桌面端态势总览，而不是详情区。它服务于“快速扫一眼场上还有谁、谁死了、谁是什么身份”。

### 4. 当前回合发言记录独立成区

在远离中央时间线的一侧增加一块独立的 `SpeechLedger` 区域，只展示当前回合的玩家发言：

- 数据范围：`message.round === currentRound && message.type === "player"`
- 排序：沿用消息原始顺序
- 每条记录展示：说话人、轮次/阶段标签、正文内容
- 顶部摘要展示：本轮发言数、最近发言人

这块区与时间线同步更新，但只回答一个问题：**“这轮都说了什么？”**

### 5. 详细席位卡片降级为次级视图

现有 `PlayerGrid` 仍保留，但不再贴近中央舞台。设计上将其降级为次级信息区，例如：

- 中央舞台下方的整块区域
- 或者页面较低位置的扩展区

这样既保留完整信息，又不会和环形席位略缩图发生视觉重复。

## Layout Structure

桌面端结构建议如下：

```text
status-strip

┌──────────── left support ────────────┬──────── center stage ────────┬────── right support ──────┐
│ compact state / latest status        │ narrative timeline           │ current-round speech log   │
│ optional small focus card            │ ring of seat nodes around it │ speech count / latest      │
└──────────────────────────────────────┴───────────────────────────────┴─────────────────────────────┘

player detail grid (secondary section)
```

实现上不要求真的把圆环和时间线做成单个复杂 SVG；更稳妥的是：

- 用一个中心容器放时间线
- 用绝对定位的轻量席位节点围绕中心容器
- 用独立右栏放发言记录

这样可控、易测，也能在不引入复杂图形系统的前提下完成视觉目标。

## Data Model Changes

### New View-Model Shape

在现有 `GameViewModel` 基础上新增两组派生数据：

1. `speechLedger`
   - `items`: 当前回合玩家发言记录数组
   - `count`: 记录数
   - `latestSpeaker`: 最近发言玩家名或 `null`

2. `seatRing`
   - 每个节点包含：`seat`, `name`, `role`, `alive`, `team`
   - 该数据可以直接从 `players` 派生，不需要单独请求

### Why View-Model Derivation

这两块都来自现有 `state + messages`，属于展示模型，而不是新的领域状态。把它们放在 view-model 层有两个好处：

- 保持组件层只负责渲染，不重复过滤逻辑
- 可以用纯函数测试覆盖发言记录与席位节点派生行为

## Component Changes

- `GameShell`
  - 改为桌面端稳定内容网格，不再让中段主区被 `situation` 压缩为零高度
  - 为中央舞台、侧向支持区、次级席位详情区提供清晰容器边界

- `PhaseHero` 或其替代组件
  - 收缩成状态条式组件，减少垂直占位

- `NarrativeLog`
  - 保留核心职责
  - 可能调整外层尺寸策略，使其更适合作为中央固定舞台

- `SpeechLedger`（新增）
  - 使用现有 shadcn `Card`、`Badge`、`ScrollArea`
  - 负责当前回合玩家发言展示

- `SeatRing`（新增）
  - 使用轻量席位节点，强调状态而不是详情

- `PlayerGrid`
  - 作为次级区域继续存在

## Styling Approach

- 复用现有 shadcn 组件与项目已安装的 `card`、`badge`、`scroll-area`
- 不覆盖基础组件的颜色和排版语义，布局通过 `className` 完成
- 死亡态优先使用语义化弱化样式，如低透明度、低对比边框、去高亮
- 圆环节点尽量轻量，避免引入复杂动画和高维护成本的自定义图形

## Risks And Tradeoffs

### Risk: 圆环席位过于花哨，影响可读性

控制措施：只做桌面端轻量缩略节点，不放大量文本，不塞完整卡片。

### Risk: 详细席位与圆环席位信息重复

控制措施：明确职责分层。圆环用于快速态势，总览卡片用于详细身份与状态确认。

### Risk: 移动端被桌面重构带坏

控制措施：新布局条件严格限制在桌面断点；移动端继续走现有纵向堆叠结构。

### Tradeoff: 当前回合发言记录与时间线存在重复内容

这是有意设计。时间线回答“发生了什么”，发言记录回答“这轮都说了什么”。两者关注点不同。

## Testing Strategy

### Unit / Component

- `view-model.test.ts`
  - 派生当前回合玩家发言记录
  - 忽略非当前回合或非 `player` 消息
  - 正确生成最近发言人摘要

- `game-shell.test.tsx`
  - 更新对桌面布局类约束的断言，避免继续依赖 `lg:h-screen` 这种会压坏内容的结构

- 新增席位略缩图组件测试
  - 活着与死亡节点样式分化
  - 节点能展示席位号与身份

### Browser Validation

- 桌面端打开控制台后，主舞台、环形席位、发言记录、详细席位区同时存在且无零高度容器
- 推进阶段后，时间线与“当前回合发言记录”同步变化
- 死亡玩家在环形席位中切换为弱化态
- 移动端不要求适配新结构，但现有纵向布局仍能正常浏览

## Implementation Notes

- 这次变更不需要新增后端接口
- 不需要本地缓存发言记录；刷新后始终由服务端消息重新派生
- 如果发现“身份”字段在圆环中视觉负担过大，可以使用更紧凑的身份缩写或图标，但不改变其需求语义
