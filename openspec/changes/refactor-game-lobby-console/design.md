## 概述

本次变更采用“舞台式大厅 + 对局控制台”方案：保留现有单页应用与服务端驱动数据流，不新增路由与本地游戏引擎；但把首页明确改成 Lobby，把进行中对局入口从“自动跳转”改成“首屏继续卡片”，并将对局页重组为更清晰的观战控制台。

## 关键决策

### 1. 首页始终先进入 Lobby

- `refresh()` 仍会在应用启动时同步服务端状态。
- 但首屏渲染不再因为存在进行中对局而直接进入 Console。
- 应用需要引入一个显式“是否进入当前对局”的前端展示状态，由用户从 Lobby 点击“继续当前对局”或“开始新游戏”触发。

### 2. 保留单入口、双主视图结构

页面仍保持单入口应用，但主视图改成：

```text
App
├─ Lobby
└─ GameConsole
```

- Lobby：开局入口、服务状态、进行中对局卡片、简短状态摘要。
- GameConsole：阶段焦点、时间线、玩家态势、操作区。

### 3. 1080p 首屏优先级高于长页展示

- Lobby 首屏只保留一个主叙事区和一个继续对局侧卡。
- Console 首屏采用“两栏 + 下方玩家区”。
- 整页滚动改为局部滚动优先，尤其时间线区域内部滚动。

### 4. 时间线成为主舞台

- 时间线节点分为：系统播报、AI 发言、投票动作/结果。
- 顶部单独突出“当前镜头/最新节点”。
- 历史节点使用统一纵向时间线展示轮次、阶段、发言者和内容。

### 5. shadcn 组件优先，减少自定义样式容器

优先使用/新增：`card`、`badge`、`separator`、`scroll-area`、`alert`、`skeleton`，必要时使用 `avatar`。

## 数据与状态流

### 保留的逻辑

- `useGameConsole` 继续负责请求协调、错误处理与后台同步。
- `view-model.ts` 继续负责从服务端状态和消息推导 UI 展示字段。
- `start / refresh / advance` 的动作边界不变。

### 新增的前端展示状态

需要新增一个轻量展示层状态，用于区分：

- `lobby`
- `console`

该状态只决定“当前显示哪个主视图”，不伪造任何游戏进度；真正的游戏状态仍完全由服务端接口决定。

## 组件重构方向

### Lobby（由现有 `PreGameScreen` 演变）

- 保留服务检查与开始游戏能力。
- 新增进行中对局卡片与摘要信息。
- 文案从“演示台 / command stage”转为正式产品语言。

### GameShell

- 从通用四槽容器改为更明确的控制台骨架。
- 保证首屏稳定落在“左时间线、右控制区、下玩家区”。

### NarrativeLog

- 升级为观战主舞台。
- 明确展示 AI 发言、投票、系统播报的视觉差异。

### 附属组件

- `ControlPanel`：收紧操作文案和层级。
- `GameSummary` / `StatusStrip`：信息去冗余，保持关键状态可见。
- `RoleSpotlight` / `PlayerGrid`：围绕当前局势而不是测试示例式陈列。

## 范围边界

本次不包含：

- 新路由、多页导航、房间系统
- 新增后端接口
- 完整历史对局列表
- 真正逐票事件数据以外的后端能力扩展

如果当前后端消息流不含完整逐票细节，则前端仍提供统一时间线结构，但对投票过程按现有数据能力降级展示。
